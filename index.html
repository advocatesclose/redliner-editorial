Thesis
On Complexity

Tools that do not degrade with scale.

Computer programmers as we know them today have existed for less than a century. They are not credentialed or licensed. They do not sit an exam.

And yet the things entrusted to computer programs are not somehow less important than the things entrusted to professionals who do. A Boeing 747 does not stay aloft because it is required by law to do so. It stays aloft because it is governed by computer programs that make it able to do so. How this is possible is worth understanding.

Over a remarkably short period, programmers have developed tools, practices, and protocols for building large-scale systems that interoperate with one another. iPhones are not less useful when three billion people have one. The tools do not degrade with scale.

Laws, by contrast, become harder to use when the number of people who must apply them increases. It is not obvious that this limitation reflects some fundamental constraint. It may simply reflect a gap in tooling.

If you print a document to Word and then print it to PDF, you have not done one thing twice. You have done two things once each.

Printing is printing. Redlining is redlining.

Architecture
01

XPC Services

Each conversion runs in a launchd-managed separate process. The system allocates and reclaims resources. Your document editor never shares memory with the export pipeline.

Multi-Process
02

Parallel Redlining

Myers diff algorithm over NSXPCConnection, coordinated by NSFileCoordinator. Ten documents redline simultaneously without contention or data races.

Structured Concurrency
03

Zero-Copy IPC

No document content crosses the process boundary. Only file paths are exchanged between processes. The XPC service reads from disk, writes to disk, and exits.

File Paths Only

Privacy is not a trade-off. It is a design constraint.

Separate processes. Separate memory. No content on the wire.

Technical Specs

No privilege without privacy.

XPC Services

NSXPCConnection, NSXPCListener, launchd lifecycle management. Each service is a separate process with its own memory space.

File Coordination

NSFileCoordinator batch prepare API for concurrent file access. Prevents data races across multiple XPC export workers.

Multi-Format Export

Pandoc over XPC for HTML, PDF, and DOCX generation. One source document, three outputs, zero manual steps.

Structured Concurrency

Swift task groups with automatic cancellation propagation. Parallel work without callback hell or manual thread management.

SwiftData

@Model persistence layer for document metadata, edit history, and transaction state. Core Data under the hood, Swift macros on top.

TextKit 2

NSTextStorage for attributed text management. Edit transactions encrypted with AES-GCM before they touch disk.

Privacy by Architecture

The architecture makes the wrong thing hard to do.

XPC services run in separate processes with separate memory. No document content crosses the process boundary — only file paths. Edit transactions are encrypted with AES-GCM before they touch disk. Privacy is enforced by the structure of the system, not by a policy that someone might override.

A single edit should propagate to every output as a matter of architecture, not effort.
